Google Analytics 4 for the GM91 landing pages

How we measure page views, button interactions, and the full conversion funnel — built on the tracking the pages already emit. gtag.js direct, no Tag Manager.

Research: Kevin Build: Blake For: Bala Issue #5 · June 2026

TL;DR

  1. The page is already GA4-ready — its track() helper pushes to dataLayer and calls gtag(). The moment we drop in a real Measurement ID, every existing event flows to GA4.
  2. 3 small code edits in audit.html (live tag + a WhatsApp-click event + the recommended generate_lead event). Same tag added to index.html.
  3. The real work is GA4 admin config: register 3 custom dimensions, mark 3 key events, build the funnel.
  4. Your two action items: a real G-XXXXXXXXXX Measurement ID and confirming the production domain. Everything else is buildable now.

1Create the GA4 property & web stream

  1. Admin → Create Property. Name it GM91; set time zone (GMT+05:30) India and currency INR.
  2. Create a Web data stream (Property → Data Streams → Web) with the production URL.
  3. Copy the Measurement ID — looks like G-XXXXXXXXXX.
  4. Confirm Enhanced Measurement is ON (default).

2Page views

page_view fires automatically on the gtag config call. For our multi-page static site, the only requirement is the same snippet with the same Measurement ID in the <head> of every page. GitHub Pages does full reloads, so each navigation fires its own page_view — no SPA handling needed. Add the snippet to both audit.html and index.html.

3Button interactions — gtag direct (not GTM)

Recommendation: gtag.js direct. The site already calls window.gtag('event', …) inside track(), so wiring gtag is a one-line uncomment. GTM would add indirection and a second script for zero benefit on a 2-page, developer-owned static site.

4Code changes in audit.html

Three small edits, all building on the existing track() helper. Don't refactor track() — it was written GA4-ready; just feed it gtag.

Edit 1 — Replace the commented placeholder with the live tag

<!-- Google Analytics 4 (gtag.js) -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXXXX"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){ dataLayer.push(arguments); }
  gtag('js', new Date());
  gtag('config', 'G-XXXXXXXXXX');
</script>

Replace both G-XXXXXXXXXX with the real ID; add the same block to index.html. This defines window.gtag — exactly what track() checks for — so all existing events light up instantly.

Edit 2 — Dedicated WhatsApp-click event (fires before the user leaves)

/* WhatsApp one-tap = primary conversion. Fire a dedicated event
   (gtag uses sendBeacon, survives page unload) before leaving for wa.me */
document.querySelectorAll('a[href^="https://wa.me/"]').forEach(function(el){
  el.addEventListener('click', function(){
    track('whatsapp_click', {cta_location: el.getAttribute('data-cta') || 'wa-link'});
  });
});

Edit 3 — Recommended generate_lead event on success

Inside showSuccess(), right after track('lead_success', {}):

/* GA4 recommended lead event — feeds the Lead acquisition report */
track('generate_lead', {currency:'INR', value:0});
No other JS changes. ab_exposure, cta_click, lead_submit, lead_success already flow correctly through track().

5Conversion funnel & key events

Mark key events

Admin → Events → toggle Mark as key event on: whatsapp_click, lead_success, generate_lead. Leave cta_click, lead_submit, ab_exposure as standard.

Build the funnel (Explore → Funnel exploration)

StepEvent
1page_view
2cta_click
3lead_submit (form path)
4lead_success / generate_lead

Use a closed funnel for true drop-off. Because WhatsApp skips the form, also build a second funnel: page_view → cta_click → whatsapp_click so the WA path isn't undercounted.

A vs B: drag ab_variant into Breakdown to split every funnel step by variant — an instant read on which converts better.

7The WhatsApp caveat

The one-tap button navigates off-site into WhatsApp. GA4 can't see what happens inside the app — we measure the intent to leave (the click), not the in-chat outcome.

  • Capture the click before they leave — the whatsapp_click event (Edit 2) via gtag's sendBeacon transport, built to deliver during page unload. Mark it a key event.
  • Free safety net: Enhanced Measurement's outbound click also records the wa.me navigation independently.
  • True closed-loop (chat → conversion) would need the WhatsApp Business API / CRM — a later phase, not this one.

8Privacy / consent (practical)

  • India DPDP Act: analytics cookies will need explicit opt-in consent, but enforcement is phased — full effect by May 2027. Safe to ship GA4 now for India-local traffic.
  • If we run ads to EU/UK: GDPR + Google Consent Mode v2 apply now.
  • Recommendation: ship GA4 now → add a footer privacy line → backlog a lightweight consent banner + Consent Mode v2 before May 2027 or before EU targeting, whichever comes first.
  • GA4 does not store full IP addresses (used transiently for geo, then discarded) — a baseline DPDP plus.
General guidance, not legal advice — get a quick legal check on the consent banner specifics before 2027.
